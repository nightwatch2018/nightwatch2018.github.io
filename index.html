JVM：类加载器浅析

发表于 2018-05-15 | 分类于 Java

这两天在知识星球上看到了一个问题，正好是我想写的一个主题，该问题如下：

最近在学习类加载器的时候遇到了一个问题，就是线程上下文类加载器，所有的资料都说是java提供标准，将接口在rt.jar中定义，但是具体实现由各大厂商实现，比如jdbc，我对此不太理解，为什么是核心类去加载具体的实现，而不是我发现一个不认识的类，去尝试用appclassloader加载，它扩展自一个不认识的接口，然后去让appclassloader加载这个接口，实际加载的是bootstrapclassloader，加载完接口再去用appclassloader去加载这个类。可能思维有些混乱。

  实际上该问题我已在星球中答复，从问题产生的原因到JDK给出的解决方案；这但还是感觉并不像写技术文章那样有条理,遂写这篇技术文章并整理思路。

  先说结论，由于在类加载器的双亲委托的机制下，根加载器无法加载厂商的接口实现类，才导致需要打破双亲机制，增加线程上下文加载器来解决这种情况，下面我们挖一挖背后的原因所在。

双亲委托机制

命名空间

我们直接给出命名空间的定义和其特征：

  所谓命名空间就是由子加载器及其所有父加载器所加载的类而构成，这其中有两点需要注意：

  1. 在同一命名空间中，不会出现类的完整名字（包括类的包名）相同的两个类。

  2. 在不同的命名空间中，有可能会出现类的完整名字（包括类的包名）相同的两个类。

  现在可以反过来再体会下上文提到的双亲委托机制下。父加载器与子加载器共同承担着类加载的工作，有的类是父加载器加载的，有的则是子加载器加载的。

  那么子加载器是委托父加载器来加载一个类的，理所当然能看到父加载器加载过的类；反过来就不一样了，父加载器看不到子加载器加载过的类，因为每个加载器都有各自的指定加载目录，父加载器看不到子加载器的目录，所以没有任何办法加载那里的类，或者说压根就不知道那里还有待加载的类。

  所以这才是问题的本源，双亲委托加载机制虽然保证了类加载的安全性，但是在一定程度上限制了其可使用场景，比如SPI接口规范。

  既然我们已经追溯到了问题本源，就可以寻找解决之道了；幸好，JDK已经为我们安排好了一切，提供了一个”后门”，也就是下面要谈到的线程上下文类加载器。

Kafka分区浅析

发表于 2018-05-13 | 分类于 中间件

分区概念

  分区概念是与主题紧密联系的，每个主题下可以有多个分区（至少有一个）。

  同一主题下的不同分区包含的消息是不同的。每个消息都会被分配一个偏移量（offset），它是消息在所在分区中的唯一编号，也是kafka用来确保在一个分区内的顺序；但offset不跨越分区使用，也就是说kafka只保证在同一个分区中的消息是有序的，而在多个分区中无法保证顺序性，如下图：

  

  每个分区都是有序的，不可变的消息序列，后续新来的消息会持续不断地添加进分区的后面，相当于一次结构化的提交日志。分区中的没一条消息都会被分配一个连续的ID，即offset，用来标识在分区中的每一条消息。

Kafka核心概念

发表于 2018-05-13 | 分类于 中间件

生产者（Producer）

  生产消息的组件，并将生成的消息发送给消息队列；可发送字符串或二进制数据。

消费者（Consumer）

  消费消息的组件，从消息队列拉取消息。

代理（Broker）

  本质上就是消息队列本身，在Kafka这个领域中，Broker指的就是一台Kafka Server。

  结合前两个概念，也就是生产者会把生成的消息发送给Borker，而消费者再从Broker拉取消息；使两者都面对Kafka编程，实现解耦的目的。

主题（Topic）

  在Kafka中属于一种逻辑概念，也就是说将消息进行分类汇总，可以看成是某一类。一个主题下可以有多条消息，那么不同主题就有各自属于它们那一类的消息。

  生产者可以指定一个主题来发送相关消息，消费者同样可以根据一个主题来拉取指定消息。

消息（Record）

  是Kfaka中的一个原子概念，指的就是生产者发送和消费者拉取的一个原子事物。

  消息由两部分组成，key+value，本质上都是字节数据。key是根据一定策略，将value存储到指定的分区上；value则是消息的真正载体。
