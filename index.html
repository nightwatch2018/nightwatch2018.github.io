Netty：官网初探

发表于 2018-05-17 | 分类于 开源框架

  首先和大家聊一聊我对学习方法论的一些思考。

  当我们去学习一个新框架的时候，一般都是去网上找几篇文章看一下或者买本书学习。我个人认为那些内容都是别人总结过后的，你得到的价值也就只是你能看到的那些，但是隐藏在背后的价值是作者本人的；包括你看我写的这篇博文也是一样，我的收获远远比你看到的价值更多，包括学习和思考能力等，这种价值我自称为隐形价值。

  我们不禁发问，要如何去积累属于自己的隐形价值？以这篇博文先开个头，呈现出我自己的方法论；也许我的方法未必适合你，这需要形成你自己的学习方法论，也就是俗称的套路，这样以后再遇到一个新框架的学习都能如法炮制，那时你积累的隐形价值会使你更加自信。

官网掠影

自我介绍

  回归正文，目前我们学习的大部分框架都会有自己的官网，通常首页就是对该框架的通盘介绍。既然我们要学习某个框架，至少要访问一次它的官网，我们可以了解一下它是什么或是能干什么，顺便捕获一下有助于我们日后准确理解这个框架的关键信息以及它的生态周边，需要注意的是，英文技术网站上的英文文法一般不是特别难，我们程序员都能看得懂，不要惧怕英文，要敢于尝试。

  我这里提取出关于Netty的自我介绍：

Netty is an asynchronous event-driven network application framework
for rapid development of maintainable high performance protocol servers clients.

  翻译过来是这样，Netty是一个异步的事件驱动的网络应用框架，用于快速开发可维护、高性能的基于协议式的服务端/客户端网络应用程序。

  主要看三个关键字，异步、事件驱动和网络应用，这是官方为Netty定义的最重要特征。

  为了更好的理解这句话，我这里引入一下英文文法，三个形容词用来修饰“框架”这个名词，那么距离该名词越近的形容词，就越能体现它的本质。

  这样我们就能更准确地理解Netty是什么？首先它是一个框架，用来帮助我们开发自己的网络应用程序，该框架的内部实现是基于事件驱动模型的，并且实现了异步的机制。

  这里简单提一下Netty中的事件驱动模型所涉及到的相关概念。

  关注分离，是软件工程中的一个架构设计方法论，就是将复杂的问题简单化，分而治之。也就是说，将系统中的变化点隔离封装出来，这样有一个点发生了变化，不会影响其他部分。在Netty中，就是将各种事件根据协议的规范抽取出来作为回调方法，监听器一旦发现某事件发生，就会主动回调这些方法。

  SEDA：Staged Event Driven Architecture，Matt Welsh博士提出的一种高性能互联网服务器架构模型。在Netty中，就是将一个请求拆分成若干个阶段，每个阶段根据自身情况可以用不同数量的线程来分别进行处理，阶段间是通过事件驱动的异步通讯模式来进行通信。每一个阶段可以配置自己的线程数量，需要合理配置。

  值得注意的是，这里的异步概念并非是AIO的那种异步机制，其实Netty5.0版本就是基于AIO实现的，但是Netty开发团队认为引入AIO极大地增加了Netty的复杂性，但是又没有看到在性能上有任何好处，还增加额外的工作量，因而放弃后继续回归到4.X的版本维护中，具体原因已在GitHub上的Issue-Remove master branch #4466说过此事，有兴趣的可以看下。

组件架构

  接下来看看官网提供的Netty组件架构图：

  一张图就能让我们知道Netty的各大组件都能实现了哪些功能。比如有传输服务的组件，Socket和HTTP Tunnel等；有支持多种网络协议的组件，HTTP、SSL、Google Protobuf等；有高性能的核心技术组件，零拷贝技术。

、  我们再来看看Netty利用这些功能，在实际应用中能完成什么，有以下三点：

  1.HTTP服务器，Netty利用自己的编码方式来实现类似于SpringMVC+Tomcat那样的功能，由于Netty自身的事件驱动模型，在性能方面肯定要比Tomcat好得多。

  那么为什么不直接替换成Netty呢？这是因为SpringMVC的在开发效率上要比Netty高得多，并且底层设施都已经构建好了，只需要程序开发人员专注业务逻辑就好，而Netty就相对来说偏底层了，并未实现Servlet规范，最主要的请求路由功能也都未提供支持，这需要我们自己去实现，比如在SpringMVC中就可以方便地通过spring的配置文件或是Annotation进行映射，然后根据请求的URI就能定位到Controller中的某个方法。

  又有一个疑问，那为什么Netty不支持这些功能呢，完全取代SpringMVC呢？开发团队可能出于某种考虑，有意将Netty定位于底层框架，这样可以让应用开发人员任意地将功能进行组合。

  2.Socket应用，比如RPC或节点间通信，Netty在这个领域内应用得非常广泛。各大开源项目在底层均使用Netty作为传输的框架，并且可以在其提供的传输机制上定义自己的协议。相比HTTP协议来说更加轻量化，传输冗余的内容（比如Header）大大减少，使传输效率大大提高。

  3.WebSocket应用，是HTML5的组成部分，客户端与服务器端一旦建立起连接，就会长期保持，互相推数据。应用场景比如有服务器端的消息推送，在线聊天或实时传输等。同Socket应用一样，仅传递对端需要的数据本身即可，而没有任何冗余的信息。

生态周边

  为了提高学习动力，我们再来看一下Netty的生态周边。Netty可是作为众多知名中间件的基础网络通信框架，比如耳熟能详的有，阿里的Dubbo和RocketMQ，Apache的Akka、Cassandra和Kafka等。

总结

  以上就是Netty官网首页展现给我们的一些有效信息，更深入的学习还需要我们去看它的User Guide；后续我们将一起学习如何使用Netty，并深挖其背后的实现原理。

Kafka分区浅析

发表于 2018-05-13 | 分类于 中间件

分区概念

  分区概念是与主题紧密联系的，每个主题下可以有多个分区（至少有一个）。

  同一主题下的不同分区包含的消息是不同的。每个消息都会被分配一个偏移量（offset），它是消息在所在分区中的唯一编号，也是kafka用来确保在一个分区内的顺序；但offset不跨越分区使用，也就是说kafka只保证在同一个分区中的消息是有序的，而在多个分区中无法保证顺序性，如下图：

  

  每个分区都是有序的，不可变的消息序列，后续新来的消息会持续不断地添加进分区的后面，相当于一次结构化的提交日志。分区中的没一条消息都会被分配一个连续的ID，即offset，用来标识在分区中的每一条消息。

Kafka核心概念

发表于 2018-05-13 | 分类于 中间件

生产者（Producer）

  生产消息的组件，并将生成的消息发送给消息队列；可发送字符串或二进制数据。

消费者（Consumer）

  消费消息的组件，从消息队列拉取消息。

代理（Broker）

  本质上就是消息队列本身，在Kafka这个领域中，Broker指的就是一台Kafka Server。

  结合前两个概念，也就是生产者会把生成的消息发送给Borker，而消费者再从Broker拉取消息；使两者都面对Kafka编程，实现解耦的目的。

主题（Topic）

  在Kafka中属于一种逻辑概念，也就是说将消息进行分类汇总，可以看成是某一类。一个主题下可以有多条消息，那么不同主题就有各自属于它们那一类的消息。

  生产者可以指定一个主题来发送相关消息，消费者同样可以根据一个主题来拉取指定消息。

消息（Record）

  是Kfaka中的一个原子概念，指的就是生产者发送和消费者拉取的一个原子事物。

  消息由两部分组成，key+value，本质上都是字节数据。key是根据一定策略，将value存储到指定的分区上；value则是消息的真正载体。
